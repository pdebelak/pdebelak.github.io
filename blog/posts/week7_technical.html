SQL Injection

Almost every website that stores any information has a database that it communicates with through SQL. When you take user input and use that to create a SQL statement on your database, though, you can open your data up to what is known as an SQL injection attack unless you take precautions.

What is SQL injection?

Let's say you have a blogging website and you allow a user to click on a link to see all of his posts. Seems reasonable, right? You might have that link run a query like select posts from posts where username='[Whatever the user's username is]' to see those posts. What if the user set his username to Peter';DROP TABLE posts;? That would delete the entire table containing all user's posts in the database. That is a disaster for any website. It is also possible to use a similar attack to see all the information in a table. Any table containing email addresses would be bad to allow out. Imagine having a table with social security numbers or credit card numbers!

Prevention

There are quite a few ways to fight against this, and it probably isn't a bad idea to use more than one. Here are a few:

  • Store the user input in placeholder variables that can only store data of a specific type. Thus a strange username like the one listed above would be rejected.
  • Prevent certain symbols from appearing in the input, specifically things that have meaning in SQL.
  • Check if the input matches a certain pattern (especially useful for telephone nubmers or email addresses).

The essential thing is to not just pass any user input directly to your database without some way of confirming that the user is not adding SQL statements of their own.

Lessons

SQL is a standardized language that virtually everyone uses. This is great! If only everything coding related was this standardized! This introduces the problem, though, of every hacker knowing exactly how to communicate with your database. By carefully paying attention to this problem, though, you should be able to avoid any SQL injection attacks.
