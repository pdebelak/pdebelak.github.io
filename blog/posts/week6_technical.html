Comparing Object-Oriented and Functional Programming

There are many programming paradigms out there and it can be hard to understand their differences, especially if all of your programming experience falls under the same paradigm. I have only worked in object-oriented languages so I wanted to take some time to research functional programming and see how the two compare.

Object-Oriented Programming

In object-oriented programming you create objects that have data and behavior embedded in them. So, you might have a Circle class that stores all the behavior and types of information that all circles have. Then, if you need a circle you create one, inputting the information specific to that specific circle. Make sense? Probably not! We will have some examples in a bit.

Functional Programming

Functional programming separates data and behavior. The way it does this is to ensure that each function (i.e. behavior) always outputs the same thing with the same input. This means that if you want to change the output, you need to input different data. Okay, now we really need some examples.

Okay, fine. Examples

We will talk about circles. In Ruby (the object-oriented language I am most familiar with), if you want to make circles and find their area you might do this:

  class Circle
    def initialize(radius)
      @radius = radius
    end
    def area
      Math::PI * @radius * @radius
    end
  end

Then we can create a circle with a given radius:

  circ = Circle.new(3)

and calling circ.area will give us the area.

How about functional programming? We'll just stick in Ruby for this, but functional programming generally is done in a function language (Haskell is popular).

  def circle_area(radius)
    Math:PI * radius * radius
  end

What happened here? How is this not the same as above? It is pretty much the same, except you notice that the function is not part of a class. It will always give the same output for the same input, whereas with the circle class we might do circ.radius = 2 and all of the sudden calling circ.area gives us a different answer. In object-oriented programming we care about the object, whereas in functional programming we care about the function. If we want to get the area of a circle with a different radius, we just call the circle_area function again with new input. It is not important if it is the same circle shrunk down or a new circle.

Neither of these approaches is better or worse, although there are certain programming situations for which one of these paradigms seems better suited. (Note: I just love the word paradigm! I feel like college professor.) Some people feel that functional programming is the way of the future, though, so it might be best to feel comfortable in both object-oriented and functional programming.
