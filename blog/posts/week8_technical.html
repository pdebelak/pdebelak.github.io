Big O Complexity and Notation

When you aren't sure of how big the input of a method or function (we'll just stick with the term method right now) is, you have to be aware of Big O Complexity.

What is Big O Complexity?

The core of Big O Complexity is looking at how quickly a method runs depending on the given input. You might have a method that works perfectly when the input is two arrays with 3 items in them each, but how does it run with 1,000,000 items in each array? That might be an extreme difference, but some methods handle that many items better than others.

Examples

Okay, let's look at some examples. Take a look at this method:

  def add(array)
    array.inject(:+)
  end

This should take more time the longer the input array is. At this point we aren't too concerned about how much extra time. Let's see a slightly different example, though.

  def add_arrays(array1, array2)
    array1.each do |num1|
      array2.each do |num2|
        num1 += num2
      end
    end
  end

This method also takes more time the longer the input arrays are. I think you see, though, that this second array can get out of hand much more quickly than the first array.

How do we talk about this complexity?

Luckily, notation has been developed to talk about these differences in complexity. This is known as Big O Notation. There are three types of Big O Complexity and they are notated like this:

  • O(1)
  • O(N)
  • O(N2)

O(1) refers to a method that takes the same amount of time regardless of the input. One example would be the #first method in Ruby. It doesn't matter how big the array is, it will always just look at the first item and return it.

O(N) refers to a method that scales in a linear manner with the input. The add method listed above is an example of that. Essentially, doubling the size of the input doubles the amount of time needed to run the method.

O(N2) refers to a method that scales in an exponential manner with the input. The add_arrays method above is of this type. Doubling the size of the input would quadruple the amount of time needed to run the method. These types of methods are ones that are very dangerous if the size of the input is unknown since the needed processing power can quickly get out of hand.

Conclusion

It is important to be aware of the Big O Complexity of your methods. Inevitably, some of your methods with scale exponentially, but it is something to keep in mind when creating a program. You don't want to write the code that makes the program run for 10 minutes with the wrong input!
